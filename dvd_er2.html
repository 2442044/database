2. 正規化の設計

第1正規化 (1NF)

一つのセルに複数の値を入れない（住所の一部や複数の電話番号を分離）。

第2正規化 (2NF)

主キーに完全従属させる。複合主キーがある場合、部分従属を排除する。

第3正規化 (3NF)

主キー以外の項目間の従属関係（推移的関数従属）を排除。今回、ジャンル名をGENRESテーブルに分離しました。これにより、ジャンル名が変更されても、一箇所の修正で全DVDに反映されます。

3. トランザクション処理 (ACID)

整合性を保つための重要な処理フローです。

貸出処理のトランザクション例:
  1. START TRANSACTION
  2. rentalsテーブルに新規レコード挿入
  3. dvdsテーブルのstock_countを1減算 (UPDATE)
  4. もし在庫が0未満になったら ROLLBACK
  5. COMMIT

4. SQLクエリ例 (Join/副問合せ)

4.1 複雑なJoin (現在の貸出一覧と会員・DVD詳細)

-- 誰が何を借りているか、ジャンル名を含めて取得
SELECT 
    u.name AS 会員氏名,
    d.title AS DVDタイトル,
    g.name AS ジャンル,
    r.rental_date AS 貸出日
FROM rentals r
JOIN users u ON r.user_id = u.user_id
JOIN dvds d ON r.dvd_id = d.dvd_id
JOIN genres g ON d.genre_id = g.genre_id
WHERE r.return_date IS NULL;

4.2 副問合せ (最も借りられているDVDのタイトル)

-- レンタル回数が平均より多いDVDを取得
SELECT title 
FROM dvds 
WHERE dvd_id IN (
    SELECT dvd_id 
    FROM rentals 
    GROUP BY dvd_id 
    HAVING COUNT(*) > (
        SELECT COUNT(*) / COUNT(DISTINCT dvd_id) FROM rentals
    )
);

5. 制約事項

  • Entity Integrity: 各テーブルのPKは非空かつユニーク。
  • Referential Integrity: rentals.user_idは必ずusersに存在しなければならない。
  • Domain Integrity: stock_countは0以上でなければならない。
